Holy Crap I Heart TypeScript

December 9, 2016

Today was officially Day 7 as a junior developer. There was this saying at bootcamp that as junior devs we wouldn’t be writing any meaningful code for the first month at the company - so I honestly had the mindset going in that there would be a lot of hand-holding and peer-programming for the first couple of weeks. But from Day 1, I was thrown head first into the deep end. My first project was an internal calculator system for a bank…whattttt? Writing in TypeScript….whattttt? Needless to say, I was terrified to make any commits and was especially terrified because the sole developer who wrote the initial code is the CTO (aka my boss).

However, he was incredibly patient; allowing me to bother him whenever I needed clarification or was stuck. It helps that he used to be a teacher and that does explain how he brought such clear explanations to his methodology and concepts. I guess it’s unfair for me to say I was thrown headfirst - I just felt a lot like this:

Brings me on to the next topic - holy crap I love TypeScript. Why has no one told me about this?! I've heard the words TypeScript, CoffeeScript, *Script used in passing but no one grabbed me by the shoulders, shook me, and said 'STOP WHAT YOU'RE DOING AND LEARN TYPESCRIPT RIGHT NOW'. I feel so cheated! I've been missing out on this experience for the last six months!!!

So to those unfamiliar - what is TypeScript? It's an open source 'language' by Microsoft and is 'an optional type system for JavaScript'. When my boss started describing my new project - I heard the word 'TypeScript' and panicked. I didn't want to look incompetent on my first day and that would be hard to prove otherwise if I had to pick up a completely new language.

Fear not folks: TypeScript == JavaScript + a layer that adds cool stuff/features!

While TypeScript has some new syntax, it ends up compiling to JavaScript and one of the features it adds on is compile-time checks on your code. When my boss described that to me, I was like, 'So it's like a linter! COOL!' And he responded, 'Yes, at it's most basic level, you could say that…’ He was right. It is much much more than that.

With TypeScript you can write with more classical object-oriented programming patterns that people familiar in Java or C# would be used to. This is the ‘optional type’ part of TypeScript. For example:

var cheese = 'bobcat';

Thats’s valid JavaScript. But in TypeScript you can do:

var cheese: string|number = ‘bobcat’;

Whattttt?

Basically that’s doing the same thing but in the TypeScript version, you’re stating that the value of cheese can only ever be a string or a number. This example is also fine:

var cheese: string|number = 23;

However the next example would immediately have an angry squiggly red line underneath it (granted you’re doing a ```tsc -w``` command that watches your file and re-compiles every time you save. If not, when you compile, an error will come up in your terminal):

var cheese: string|number = [23, ‘bobcat’];

The angry squiggly red line immediately points you to the problem, tying to the first feature I mentioned of compile-time checks. My mind was blown! Imagine all the time I could have saved instead of Sherlock Holmes-ing it through files of code for a dumb dumb error like that.

Actually I’ll do a better example:


              function area(height: number, width: number): number {
                return height * width;
              }
              area(5, 5); // will pass fine
              area('5', '5'); // will throw error

So what’s the difference between the two function calls above? Well the first function call will pass fine because we’re passing in numbers (the correct type of parameters). However, in the second function an error will occur because we’re passing in strings. TypeScript will show us, even before running the script, that we’ve forgotten to convert the user input. Why does this matter? Because what if the next function we run was expecting a number returned? It will break everything and it will take you an hour to realize - damn, I forgot to parse/number that value, why didn’t I listen and use Typescript? These are pretty basic examples but already showcases the huge potential TypeScript can bring to developing and maintaining large applications.

With the release of Angular 2, more people will be developing in TypeScript (more on that here. The article also dives deeper into why TypeScript is dopesauce) and I am pretty stoked to continue building my skills. I know there will be JS purists out there smirking at everything I’ve just written but hey - what do I know? I’m just a junior dev ¯\_(ツ)_/¯

I have already been tasked with a new project (Django and Python, and had to set up my environment using Docker containers — yikes! (maybe more on that later. I’m still a bit traumatized from the experience)) and for the time being, it’s goodbye TypeScript. You know that feeling when a waiter comes to your table with a tray of food and then realises he got the wrong table? Yes that heartbreak is the *exact* same feeling I got when I was assigned to Django. Sigh.

More Coming Soon!

November 28, 2016

I've been putting off starting a blog for some time - with job searching and learning all the tools, frameworks, algorithms and interview skills required to get said job, it's easy to put content writing aside.

That said, the first step is the hardest and I plan to fill this blog with interest pieces, projects I'm working on, challenges I come across, and explanations of concepts or tools that I myself found difficult to process - all with the aim of a) digging deeper, b) doing a better job explaining and talking about programming, and c) learning from teaching someone else/putting thought to paper.

A bit of background about myself - I'm a junior developer that went through Code Fellows, a programming bootcamp based in Seattle, specialising in the full-stack JavaScript development track. While I think Code Fellows exceeded my expectations and I had a wonderful time with the instructors and other students - I feel there is still a large gap in my knowledge of Computer Science fundamentals and even JavaScript fundamentals.

This is completely understandable - bootcamps are aimed to take you from zero knowledge to a "job ready" professional in X weeks (typically 14-20 weeks). For a while I struggled with these feelings of inadequacy - that I should know everything about JavaScript fundamentals, have OOP and functional programming concepts locked down, know my stack (MEAN) inside-out, have top-notch whiteboarding skills and still have time to sleep at night.

I took a long, deep breath and began talking to other devs and hiring managers. While they are looking for the best person to join their teams, they understand my background and experience, and they value my potential higher than what I can do at a whiteboard right there and then. I definitely think that's the key that took my stress levels way down.

I am a junior developer that did 180 on my career path. While my skills should be taken at face value, this career transition and how far I've come within a year, proves I have a high capacity for learning and that I am incredibly passionate about what I do.

^ That is what I needed to believe. That is what I needed to own. Sure enough - I got a job offer at the next job I interviewed at.
